# Capability & Skills System — Design

**Date:** 2026-06-30
**Status:** Proposed
**Issue:** #24
**UI prototype:** `docs/prototypes/2026-06-30-capabilities-skills.html` (token-only, lifts into `client/styles/capabilities.scss`)

## Summary

Model the consultancy's service offering as a two-tier taxonomy and use it to match
supply (engineers) to demand (projects).

| Tier | Means | Shape |
|---|---|---|
| **Capability** | what we deliver for a customer | reference entity (Cloud Platform Engineering, Payments Integration) |
| **Skill** | what a consultant knows | reference entity (Kubernetes, Stripe API) |
| **Composition** | which skills make up a capability | many-to-many, weighted |

Engineers carry a per-skill experience level over time; projects declare the
capabilities they require over a span; the project view shows required vs. the
current team's coverage; and a recommender names the engineer best placed to fill a
gap — including growth pairings.

Three deliverables, three PRs:

1. **Taxonomy + assessments** — the four new tables, the engineer skill matrix.
2. **Demand + visualization** — project capability requirements, the coverage bars.
3. **Recommendation** — the gap-filling assignment recommender.

## Experience scale

Per engineer, per skill. The rubric every read and the recommender share:

| Level | Meaning |
|------:|---------|
| 0 | none |
| 1 | learning |
| 2 | can do with supervision |
| 3 | independently capable |
| 4 | expert / can teach |

Rendered on the sequential `--lvl-*` ramp already in `theme.scss`.

## Decisions (resolving the issue's open questions)

| # | Question | Decision |
|---|---|---|
| 1 | Rollup aggregation | **Weighted average**: `Σ(level × weight) / Σ(weight)` over a capability's skills, as-of. Per-skill detail shown alongside. |
| 2 | Taxonomy temporality | **Temporal**: `capability`, `skill`, and `capability_skill` are versioned facts read as-of. The catalog evolves — a capability defined last year has no row two years ago, so a past rail date presents only the capabilities/skills offered then; renames and re-weightings keep history. |
| 3 | Capability requirement shape | **Both**: a `target_level` (0–4 proficiency threshold) and a `quantity` (how many engineers must meet it), over a bounded span. |
| 4 | Level 0 | **Absence = 0** for reads/rollup; an explicit `AssessSkill … level 0` records a deliberate "assessed, none". Both render as 0. |
| 5 | Recommender | **Ready-now fit + mentorship pairings**: rank engineers whose rolled-up proficiency covers a gap, and surface learner+teacher growth pairs. |
| 6 | Mapping detail | **Per-skill integer weight (1–3) now**; no per-skill minimums (deferred). |
| 7 | Permissions | **Dedicated keys**: `skills.manage` (taxonomy + composition), `skills.assess` (record assessments, manager/owner). Project capability requirements stay under `project.manage`. |

A new decision record **ADR-049** (capability/skill model + weighted-average rollup
semantics) is appended to `docs/DECISIONS.md` in Phase 1. (Note: the issue named this
ADR-047, which is already taken; the next free number is 049.)

## Data model

Four new tables. Each new concept follows the established per-concept CQRS split
(`command.gleam` / `view.gleam` / `http.gleam` / `sql/` on both `shared` and
`server`), a shared `Command` envelope variant, typed `Fact`(s), a `repository`
persistence arm, and Squirrel queries.

### Reference taxonomy (temporal)

Anchors mint once; everything that can change over time is a versioned fact read
as-of — the same anchor + `daterange WITHOUT OVERLAPS` shape used elsewhere. A
capability/skill is "in the catalog as-of a date" exactly when a profile row covers
that date, so existence, naming, and offering window are one fact.

```sql
CREATE TABLE capability (id int GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY);
CREATE TABLE capability_profile (
  capability_id   int  NOT NULL REFERENCES capability(id),
  name            text NOT NULL,
  summary         text NOT NULL DEFAULT '',
  defined_during  daterange NOT NULL,
  audit_id        bigint,
  CONSTRAINT capability_profile_no_overlap
    PRIMARY KEY (capability_id, defined_during WITHOUT OVERLAPS)
);

CREATE TABLE skill (id int GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY);
CREATE TABLE skill_profile (
  skill_id       int  NOT NULL REFERENCES skill(id),
  name           text NOT NULL,
  summary        text NOT NULL DEFAULT '',
  defined_during daterange NOT NULL,
  audit_id       bigint,
  CONSTRAINT skill_profile_no_overlap
    PRIMARY KEY (skill_id, defined_during WITHOUT OVERLAPS)
);

CREATE TABLE capability_skill (
  capability_id   int NOT NULL REFERENCES capability(id),
  skill_id        int NOT NULL REFERENCES skill(id),
  weight          int NOT NULL CONSTRAINT capability_skill_weight_check CHECK (weight BETWEEN 1 AND 3),
  composed_during daterange NOT NULL,
  audit_id        bigint,
  CONSTRAINT capability_skill_no_overlap
    PRIMARY KEY (capability_id, skill_id, composed_during WITHOUT OVERLAPS)
);
```

- Anchors minted by `repository.create_capability` / `create_skill` (mirrors
  `create_engineer`): an `*_next_id.sql` + `*_create.sql` pair returning a typed
  `CapabilityId` / `SkillId`.
- `capability_profile` / `skill_profile` are versioned exactly like `engineer_role`:
  define/rename via the open-ended **change** write (`FOR PORTION OF … FROM date TO
  NULL`), retire via a bounded close. Read as-of (`defined_during @> $date`); current
  catalog via an `upper_inf` view.
- `capability_skill` is versioned per `(capability_id, skill_id)`: add / re-weight is
  the change write, remove is a close. The rollup as-of a date reads the
  period-correct weights.
- No PERIOD-FK containment: the taxonomy has no parent run (unlike `employment` /
  `project_run`), so these are versioned facts without containment — the same shape
  as the existing profile facts.

### Temporal assessments — mirror `engineer_role`

```sql
CREATE TABLE engineer_skill (
  engineer_id    int NOT NULL,
  skill_id       int NOT NULL REFERENCES skill(id),
  level          int NOT NULL CONSTRAINT engineer_skill_level_check CHECK (level BETWEEN 0 AND 4),
  assessed_during daterange NOT NULL,
  audit_id       bigint,
  CONSTRAINT engineer_skill_no_overlap
    PRIMARY KEY (engineer_id, skill_id, assessed_during WITHOUT OVERLAPS),
  CONSTRAINT engineer_skill_within_employment
    FOREIGN KEY (engineer_id, PERIOD assessed_during)
    REFERENCES employment (engineer_id, PERIOD employed_during)
);
```

`AssessSkill(engineer_id, skill_id, level, effective)` is an **open-ended change from
a date onward** — exactly `Promote`/`engineer_role`. The write is the delete-then-
insert "Change" upsert (`FOR PORTION OF assessed_during FROM $effective TO NULL` for
that `(engineer_id, skill_id)`, then insert `[effective, upper(employment))`). Read
as-of via `assessed_during @> $date`; full timeline via `upper_inf(...) AS ongoing`.
No `_current` view (consistent with `engineer_role`).

### Temporal demand — mirror `project_requirement`

```sql
CREATE TABLE project_capability (
  project_id      int NOT NULL REFERENCES project(id),
  capability_id   int NOT NULL REFERENCES capability(id),
  target_level    int NOT NULL CONSTRAINT project_capability_target_check CHECK (target_level BETWEEN 0 AND 4),
  quantity        numeric(4,2) NOT NULL CONSTRAINT project_capability_quantity_check CHECK (quantity > 0),
  required_during daterange NOT NULL,
  audit_id        bigint,
  CONSTRAINT project_capability_no_overlap
    PRIMARY KEY (project_id, capability_id, required_during WITHOUT OVERLAPS),
  CONSTRAINT project_capability_within_run
    FOREIGN KEY (project_id, PERIOD required_during)
    REFERENCES project_run (project_id, PERIOD active_during)
);
```

`SetProjectCapability(project_id, capability_id, target_level, quantity, valid_from,
valid_to)` is the bounded **clear-then-set** per `(project_id, capability_id)` — the
two-statement `project_requirement` write.

### New commands, facts, and ids

All writes carry an effective date (the change/close conventions), as elsewhere in
the app.

| Aggregate | Commands | Permission |
|---|---|---|
| `capability` | `CreateCapability`, `DefineCapability` (name/summary from a date), `RetireCapability`, `SetCapabilitySkill` (weight from a date), `RemoveCapabilitySkill` | `skills.manage` |
| `skill` | `CreateSkill`, `DefineSkill`, `RetireSkill` | `skills.manage` |
| `engineer_skill` | `AssessSkill` | `skills.assess` |
| `project_capability` | `SetProjectCapability` | `project.manage` |

Typed ids `CapabilityId`, `SkillId` in `fact.gleam`. Facts: `CapabilityProfile`,
`CapabilitySkillSet`, `CapabilitySkillRemoved`, `SkillProfile`,
`EngineerSkillAssessed`, `ProjectCapabilityRequired`.

## Rollup

Engineer→capability proficiency is the weighted average of the engineer's skill
levels across the capability's constituent skills, as-of a date:

```
proficiency(engineer, capability, date)
  = Σ (level(engineer, skill, date) × weight(capability, skill))
  / Σ (weight(capability, skill))
```

`level` coalesces a missing assessment to 0. Computed in SQL joining `engineer_skill`
(as-of) × `capability_skill` (as-of, so period-correct weights). Displayed to one
decimal; the per-skill levels back it.

## Read models & UI

All as-of the global rail date. Markup mirrors existing atoms; new rules land in
`client/styles/capabilities.scss`. See the prototype for the rendered design.

1. **Engineer skill profile** (People detail, new "Skills" tab) — the per-skill
   matrix (skill → level on the ramp, with the capabilities each skill feeds), a
   capability rollup panel, and recent assessments. "Assess skill" launches a modal.
2. **Skills taxonomy admin** (a Settings/Access-style page, `skills.manage`) —
   capabilities list, skills list, and the composition matrix (reuses
   `access__matrix`): each cell holds the editable per-skill **weight**; a focused
   per-capability composition editor modal offers weight steppers and add/remove.
3. **Project capability coverage** (project detail, new tab) — per required
   capability: target + quantity vs. the current team's coverage as coverage bars
   (have N at ≥ target vs. need M), gaps highlighted, with the covering engineers.
4. **Assignment recommendation** (project detail) — for unmet capability
   requirements, a ranked list of engineers by fit, each with a human-readable
   rationale, feeding a pre-filled Assign modal.

## Assignment recommendation

For a project's unmet capabilities (required − current team coverage, as-of), rank
engineers and surface top picks with a reason.

- **Fit** — rolled-up proficiency in the gap capability vs. its `target_level`.
- **Availability** — unallocated fraction as-of (existing allocation model).
- **Mentorship pairing** — a level-1/2 *learner* in the gap's skills paired with a
  level-4 *teacher* already on the team is offered as a growth assignment alongside
  ready-now experts.

Output: a ranked list, each rationale like "covers the Payments gap at 2.8; 60%
available", or "growth: learns Stripe / PCI-DSS under Priya Sharma; 50% available".

## Authorization

Two new keys wired through the five coordinated edits (catalog → policy → seed matrix
→ server enforcement → client `Permit` gating), modelled on `project.manage`:

| Key | Grants | Roles |
|---|---|---|
| `skills.manage` | create/edit capabilities, skills, composition | owner, manager |
| `skills.assess` | record engineer skill assessments | owner, manager |

Self-assessment is out of scope for now.

## Seed scenario

The base seed grows to a richer bench (~11 engineers) and a starter taxonomy so the
coverage and recommender demos read realistically:

- 4 capabilities, 14 skills, the weighted composition.
- Engineer assessments producing an interesting coverage/gap picture.
- `project_capability` on **Ledger Migration** with a visible **Payments gap**
  (one engineer at ≥ target, two required) to drive the recommender.

## Non-goals

- Per-skill minimum levels in the composition (weights only for now).
- Self-service assessment; assessment approval workflows.
- Heat-matrix coverage visualization (coverage bars chosen).
