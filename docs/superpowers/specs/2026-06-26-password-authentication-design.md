# Password Authentication — Design

**Date:** 2026-06-26
**Status:** proposed (implementation in progress)
**Supersedes the relevant part of:** ADR-035 "Demo identity gate, not real authentication"

## Goal

Replace the password-less demo identity gate with **real, password-based
authentication** over the existing signed-cookie session, following best
practices:

- credentials verified against hashed passwords (PBKDF2-HMAC-SHA512 — see Decisions);
- a **separate "remember me" opt-in** controlling session vs. persistent cookie;
- **logout clears the cookie** server-side;
- cookie carries `HttpOnly`, `SameSite=Lax`, and `Secure` (except localhost http).

It deliberately stays **simple**: a stateless, signed cookie — the model the app
already uses — gated now by a real credential check. We do *not* build DB-backed
server sessions, rotating remember-me tokens, theft detection, registration, or
password-reset. Those are noted as future work, not scope.

## What stays the same (the seam ADR-035 left)

The signed session cookie still carries `actor|role`; `auth.authorize` still gates
the 24 commands on `principal.role`; the journal `actor` is still derived from the
session, never the request body. Authentication changes *how the cookie is issued*,
not what it means downstream. Per-request session validation stays stateless (no DB
round-trip): the cookie is HMAC-signed, so a verified payload is trusted.

## Architecture

### 1. Storage — `account` table (new migration)

Auth credentials are infrastructure, not business-temporal facts, so this is a
plain table (not an edit-grouped temporal fact table):

```sql
CREATE TABLE account (
  id            int GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  username      text NOT NULL UNIQUE,
  display_name  text NOT NULL,
  role          text NOT NULL CHECK (role IN ('admin', 'ops', 'engineer')),
  password_hash text NOT NULL,
  created_at    timestamptz NOT NULL DEFAULT now()
);
```

`display_name` is the existing actor string (`"Priya Sharma"`, `"Admin"`, …), so
authorization and the journal are unchanged. `username` is the login id (email).

### 2. Domain — `tempo/server/account/`

CQRS convention. Authentication is a read + verify:

- `account/sql/account_by_username.sql` → `SELECT display_name, role, password_hash WHERE username = $1`
- `account/sql/account_upsert.sql` → idempotent insert (`ON CONFLICT (username) DO NOTHING`), used only by the dev seed
- `account/sql.gleam` (squirrel-generated)
- `account/view.gleam` — `authenticate(db, username, password) -> Result(Principal, AuthnError)`:
  looks up the row, maps `role`, verifies the password with `password.verify`. Errors:
  `UnknownUser | BadPassword | CorruptAccount | StoreError(QueryError)`.
- `account/password.gleam` — PBKDF2-HMAC-SHA512 `hash`/`verify` over a tiny
  `tempo_password_ffi.erl` wrapper around OTP's `crypto:pbkdf2_hmac`. Stored hash is
  a self-describing string `pbkdf2-sha512$<iterations>$<salt-b64>$<key-b64>`; verify
  recomputes under the stored salt/iterations and compares in constant time
  (`crypto.secure_compare`). 210k iterations (OWASP floor).
- `account/seed.gleam` — the dev cast (`dev_accounts()`, `dev_password`): the five
  principals with email usernames, all sharing one documented dev password. Imported
  by the seeder **and** the gleam test login helper so they never drift.

### Decision: PBKDF2, not Argon2id

Argon2id (the first-choice KDF) ships for Gleam as `argus`, whose `jargon` NIF fails
to link on this arm64/OTP-28 toolchain (the macOS `-undefined suppress`/`-flat_namespace`
flags the modern linker ignores). Rather than bake a platform-specific linker flag
into every build (and risk CI), we use **PBKDF2-HMAC-SHA512 via OTP's built-in
`crypto`** — no native compilation, builds identically everywhere, OWASP-acceptable
at a high iteration count, and a vetted KDF rather than rolled crypto. The encoded
hash records its algorithm + cost, so a future move to Argon2 (if the NIF builds) can
re-hash on next login without a schema change.

### 3. Auth module — `tempo/server/auth.gleam`

- Keep `Role`, `Principal`, `AuthzError`, `authorize`, `is_financial`, `command_tag`.
- `to_session(principal) -> "actor|role"` unchanged.
- `from_session(session)` no longer consults a hardcoded registry (accounts now live
  in the DB). It parses `actor|role`, maps the role string, and requires a non-empty
  actor. Because the cookie is signed, that is sufficient and needs no DB read.
- Add `role_from_string`. **Remove** `principals()` and `lookup()` (login now queries
  the DB; the client no longer needs a hardcoded identity list).

### 4. Web session — `tempo/server/web/session.gleam`

Owns the cookie. Hand-builds the `Set-Cookie` header so a true **session cookie** is
possible (wisp's `set_cookie` always emits `Max-Age`):

- `issue(response, request, principal, remember: Bool)` — sign the payload with
  `wisp.sign_message(request, <<payload:utf8>>, crypto.Sha512)` (identical to wisp's
  own `Signed` cookie, so `wisp.get_cookie(_, _, Signed)` still verifies it), then set
  the cookie with `cookie.defaults(scheme)` and `max_age = Some(30 days)` when
  remember is on, `None` (session cookie) when off.
- `clear(response, request)` — same cookie, value `""`, `max_age = Some(0)` (emits
  `Expires=epoch` + `Max-Age=0`).
- `principal(request)` — unchanged (`wisp.get_cookie(Signed)` → `auth.from_session`).
- `scheme(request)` — replicates wisp's localhost rule so the cookie is **not**
  `Secure` over `http://127.0.0.1`/`localhost` (otherwise the browser drops it and
  e2e login breaks).

### 5. HTTP handlers

- `web/login.gleam` — `POST /api/login` body `{username, password, remember_me?}`.
  Verify via `account.authenticate`; on success issue the session (honouring
  `remember_me`) and echo `{actor, role}`. **Uniform 401** for unknown user / bad
  password / corrupt account (no oracle); `StoreError` → 500/503. Drop
  `TEMPO_AUTH_DEMO`.
- `web/logout.gleam` (new) — `POST /api/logout`, clears the cookie, returns `{}`.
- `web/router.gleam` — add `["api", "logout"] -> logout.handle(...)`.

### 6. Client — `client/`

- `client/api.gleam` — `login(username, password, remember, to_msg)` posts the new
  body; add `logout(to_msg)`.
- `client/app.gleam` — replace the identity-button gate with a real form: email +
  password inputs (associated `<label>`s so they're reachable by accessible name),
  a "Remember me" checkbox, a "Sign in" button, and an inline error on 401. New form
  state on the model + messages (`LoginUsernameChanged`, `LoginPasswordChanged`,
  `LoginRememberToggled`, `LoginSubmitted`, `LoginReturned`). `SignedOut` now also
  fires `api.logout` (fire-and-forget) before returning to the gate.

### 7. Seeding

`tempo/seed.run` gains an idempotent `seed_accounts` step (Argon2-hash the dev
password, upsert each account) that runs **outside** the `already_seeded` guard, so
accounts are backfilled even on an already-seeded dev DB. `base_seed.sql` is
untouched (it stays frozen for e2e). `bin/seed` therefore: migrate → seed accounts →
(skip base if present). No financials are touched, so `gleam test` stays on its
0-invoice base.

## Testing

- **gleam unit tests**: `account_test` (authenticate happy path, unknown user, bad
  password); update `auth_test` (drop registry tests, keep `authorize`, keep session
  round-trip, add `role_from_string`); update `api_test` `sign_in` helper to post
  `{username, password}` mapped from the actor via `account/seed` (all existing
  operation/authorization tests keep working unchanged); add login-rejects-bad-password
  and logout-clears-session web tests.
- **e2e (Playwright)**: update `signInAs` to fill email/password (mapping the identity
  name → seeded username + dev password) and click "Sign in" — every existing spec
  keeps working through the same helper. Add a spec: wrong password shows an error and
  does not enter; remember-me unchecked issues a session cookie, checked issues a
  persistent one; logout returns to the gate and clears the cookie.

## Out of scope (future work)

DB-backed/revocable sessions, rotating remember-me tokens with theft detection,
self-service registration, password change/reset, rate-limiting/lockout, MFA. The
stateless signed cookie means logout clears the client cookie but cannot revoke a
copy already exfiltrated before it expires — an accepted tradeoff for an internal
tool, documented here.
